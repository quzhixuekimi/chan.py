### ⚠️ V8_BYMA 异常详情 - cycle_id孤立问题

**异常发现**：
- 在某些交易中发现 `cycle_id=5` 在trade_trace文件中无对应记录
- 这表示该周期的开始信息未被完整记录

**影响范围**：
- 少量交易受影响（占比 < 5%)
- 不影响单笔交易的盈亏计算
- 可能导致周期级别的分析不完整

**根本原因推测**：
1. 数据导出时序问题：周期关闭时刻与trace记录时刻的不一致
2. 极端行情下的边界情况：快速连续切换周期时可能跳过记录
3. 可接受的数据延迟：不影响交易执行，仅影响事后分析

## 🔧 问题修复记录 (2026/04/23 后续)

### v8_checks.py 文件匹配逻辑修复

#### 问题描述
初次运行一致性检查时，v8策略报告 cycle_id=5 在 trade_trace 中缺失的异常。

#### 根本原因分析
原代码 `v8_checks.py` 第23行和第38行：
```python
tfile = trades[0]      # 第一个文件: AAPL_1d_trades_v8_byma.csv
trace = traces[0]      # 第一个文件: TSLA_1d_trade_trace_v8_byma.csv ← 不匹配！
```

代码简单地取第一个 trades 文件和第一个 trace 文件，但它们不是对应的：
- trades 第一个是 `AAPL_1d`
- trace 第一个是 `TSLA_1d`

然后对比 AAPL 的 cycle_id=5 和 TSLA 的 trace（没有 cycle_id=5），导致误报。

#### 修复方案
修改 `consistency_check/v8_checks.py`：
1. 从 trades 文件名提取 `symbol_timeframe`（如 `AAPL_1d`）
2. 构造对应的 trace 文件名（`AAPL_1d_trade_trace_v8_byma.csv`）
3. 遍历所有 trades 文件，检查时使用匹配的文件对

#### 修复后结果
```json
"v8": {
  "ok": true,
  "failures": [],
  "n_trades": 1932
}
```

#### 结论
- **这不是回测策略代码的bug**，而是**一致性检查脚本的误报**
- 回测引擎 `backtest_engine.py` 的逻辑是正确的
- FORCED_LAST_BAR 会正常写入 trace
- 只需修复检查脚本的文件匹配逻辑
